# Architecture

**Analysis Date:** 2026-03-15

## Pattern Overview

**Overall:** Event-driven microservice with persistent daemon and file-watching scheduler.

### Key Characteristics:

- Express REST API server (port 9000) as central broker

- SQLite FTS5 database for full-text indexing and graph storage

- Node-cron task scheduler for periodic analysis and maintenance

- File watcher (chokidar) for real-time change detection

- Multi-processor pipeline for format conversion and enrichment

- Recursive document relationship graph using SQLite CTEs

- Hook-based integration with Claude Code and external systems

## Layers

### Daemon Layer:

- Purpose: Long-running Express HTTP server handling all API requests

- Location: `daemon/server.mjs`, `daemon/scheduler.mjs`, `daemon/watcher.mjs`

- Contains: REST endpoints, request routing, authentication, response formatting

- Depends on: Database (better-sqlite3), processors, graph module

- Used by: Claude Code hooks, REST clients, RootDispatcher

### Processor Layer:

- Purpose: Transform document content (markdown, PDF, DOCX, diagrams) into indexable form

- Location: `processors/*.mjs` (markdown, PDF, word, keyword, tree, mermaid, relink)

- Contains: Format-specific parsers, enrichment logic, FTS indexing, relationship detection

- Depends on: File system, database, markdown-it, gray-matter, natural.js

- Used by: Daemon (via hooks), scheduler (periodic runs), CLI scripts

### Graph Layer:

- Purpose: Build and query document relationship networks using recursive traversal

- Location: `graph/relations.mjs`

- Contains: Relationship inference engine, recursive CTE queries, path resolution

- Depends on: Database, document processor results

- Used by: Daemon endpoints, processors, analysis scripts

### Database Layer:

- Purpose: Persistent storage of documents, relationships, metadata, scan history

- Location: `scripts/db/schema.sql`, `scripts/db/init-database.mjs`

- Contains: 20+ tables, FTS5 virtual table, triggers, views, indexes

- Depends on: SQLite (better-sqlite3)

- Used by: All layers

### CLI/Scripting Layer:

- Purpose: Batch operations and one-off analysis tasks

- Location: `scripts/*.mjs` (scan, index, fix, validate, analysis)

- Contains: Repository scanning, batch indexing, markdown fixing, pattern detection

- Depends on: Processors, database, configuration

- Used by: Manual execution, cron jobs, one-time migrations

## Data Flow

### Document Ingestion:

1. File discovered: watcher or scan script

2. Processor routes by file type (markdown → markdown-processor.mjs, DOCX → word-processor.mjs, PDF → pdf-processor.mjs)

3. Processor extracts content, frontmatter, metadata, content_hash

4. Markdown processor detects category from path/filename/content

5. FTS index updated via database trigger

6. Relationship builder (graph/relations.mjs) finds links and creates edges

7. Keyword processor extracts TF-IDF terms, stores in keywords table

### Search Query:

1. Client: `GET /search?q=topic&repo=DocuMind&category=api`

2. Server parses parameters, builds WHERE clauses

3. FTS5 MATCH query executed with snippet generation

4. Results ranked by relevance (FTS rank)

5. Response: matched documents with highlighted snippets

### Diagram Curation:

1. Diagram generated by tree-processor or mermaid-processor

2. Auto-generated FigJam URL stored in diagrams table

3. Diagram marked pending_relinks view until curated

4. User curatesURL via `/diagrams/relink` endpoint or diagram-curated hook

5. relink-processor.mjs updates diagrams table

6. Propagates old→new URL across all 14+ repositories via propagateRelinkAllRepos()

7. DIAGRAM-REGISTRY.md updated with new curated URL

8. Trigger updates stale flag when source_hash differs

### Scheduled Analysis:

1. Scheduler.mjs cron jobs fire:

   - Every 15 min: heartbeat + stats update

   - Every 1 hour: incremental scan (changed files only via content_hash)

   - Daily 2 AM: full scan + similarity detection + deviation analysis

   - Weekly Sun 3 AM: PDF re-index + keyword refresh + graph rebuild

2. Scan determines new/modified/deleted documents

3. Processors enrich documents

4. Graph builder updates relationships

5. Deviations and similarities tables populated

6. Statistics table updated for dashboard

### State Management:

- **Documents table:** Single source of truth for file metadata and content

- **content_hash:** Enables incremental scanning (SHA-256, triggers FTS re-index only if changed)

- **scan_history:** Audit trail of all scans, duration, error tracking

- **Query results:** Cached in query_cache table with TTL triggers

- **Pending relinks view:** Dynamic view of diagrams awaiting curation

- **Stale diagrams:** Tracked via source_hash comparison in diagrams table

## Key Abstractions

### Document:

- Purpose: Unified representation of any text file (markdown, converted PDF, DOCX)

- Examples: `documents` table primary entity

- Pattern: Store raw content + parsed frontmatter + auto-detected metadata; index content via FTS5

### Relationship:

- Purpose: Directional edge in document graph (imports, parent_of, variant_of, supersedes, etc.)

- Examples: `doc_relationships` table, 8 relationship types

- Pattern: Detect relationships via content analysis (link detection, dispatch parsing, version patterns); store with weight and metadata for ranking

### Processor:

- Purpose: Pluggable format handler with standardized input/output

- Examples: `markdown-processor.mjs`, `pdf-processor.mjs`, `word-processor.mjs`, `keyword-processor.mjs`

- Pattern: Receive file path → extract + enrich → return { content, frontmatter, metadata } → insert into documents table

### Diagram Registry:

- Purpose: Bidirectional sync between on-disk DIAGRAM-REGISTRY.md and diagrams table

- Examples: `relink-processor.mjs` (propagateRelink, syncRegistryFromDb, reverseSyncFromRegistry)

- Pattern: Parse markdown table → detect URL changes → propagate via find-replace across repos → re-index markdown

## Entry Points

### HTTP Server:

- Location: `daemon/server.mjs` (Express app.listen() at line 364)

- Triggers: `npm run daemon:start` (PM2) or `npm run daemon:dev` (foreground)

- Responsibilities: Accept HTTP requests, dispatch to handlers, return JSON responses

### Scheduled Tasks:

- Location: `daemon/scheduler.mjs` (initScheduler function)

- Triggers: Cron jobs registered on daemon startup

- Responsibilities: Execute periodic scans, analysis, cleanup at fixed schedules

### File Watcher:

- Location: `daemon/watcher.mjs` (initWatcher function)

- Triggers: chokidar watches directories on daemon startup

- Responsibilities: Detect file changes, trigger incremental re-indexing

### CLI Scripts:

- Location: `scripts/scan-all-repos.mjs`, `scripts/fix-markdown.mjs`, etc.

- Triggers: `npm run scan`, `npm run fix`, etc.

- Responsibilities: Batch operations outside daemon context

### Claude Hooks:

- Location: `daemon/hooks.mjs` (processHook function)

- Triggers: POST `/hook` endpoint with event type

- Responsibilities: Route events (post-write, post-commit, diagram-curated, scan, convert) to appropriate handler

## Error Handling

**Strategy:** Try-catch at processor level; scan errors logged but don't halt batch operations; database transactions rolled back on constraint violations.

### Patterns:

- Scan failures captured in scan_history.error field with status='failed'

- Hook handler returns { status: 'error', error: message } on failure

- File access errors logged to console but continue with next file

- Database foreign key constraint errors silently ignored on upserts (INSERT OR IGNORE)

- API endpoints return 400 for missing params, 404 for not found, 500 for server errors

## Cross-Cutting Concerns

**Logging:** Console.log with [module-name] prefix (e.g., `[scheduler]`, `[graph]`, `[hooks]`); PM2 logs captured in data/logs

**Validation:** Zod schemas in config (zod is dependency); markdown-it for syntax validation; markdownlint for style rules

**Authentication:** None in v2.0 (assumes trusted internal network; port 9000 not publicly exposed)

**Caching:** Query cache table with auto-expiry trigger; FTS5 index maintained via database triggers

**Transactions:** batch() transactions wrap relationship building to ensure atomic writes; single-row operations use implicit transaction
