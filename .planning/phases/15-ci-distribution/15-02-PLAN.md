**1. Overview paragraph** explaining that DocuMind images are published to `ghcr.io/design-dvw/documind` and that automated publishing happens via GitHub Actions on version tags, but manual publishing is also supported.

**2. Automated Publishing subsection:**
- Explain that pushing a semver tag triggers the workflow: `git tag v3.2.0 && git push origin v3.2.0`
- Note the image appears at `ghcr.io/design-dvw/documind:3.2.0` and `:latest`
- Mention the workflow file location: `.github/workflows/publish.yml`
- Note: ARM64 builds may take 15-40 minutes under QEMU emulation

**3. Manual Publishing subsection** with these steps:
- Step 1: Create a GitHub PAT with `write:packages` scope at github.com/settings/tokens
- Step 2: Authenticate — `echo $CR_PAT | docker login ghcr.io -u YOUR_GITHUB_USERNAME --password-stdin`
- Step 3: Build and push multi-arch — `docker buildx build --platform linux/amd64,linux/arm64 --tag ghcr.io/design-dvw/documind:VERSION --tag ghcr.io/design-dvw/documind:latest --push .`
- Step 4: Verify — `docker buildx imagetools inspect ghcr.io/design-dvw/documind:VERSION`

**4. Pulling the Image subsection:**
- `docker pull ghcr.io/design-dvw/documind:latest`
- `docker run -d -p 9000:9000 ghcr.io/design-dvw/documind:latest`
- Note: for full usage, use docker-compose.yml with volumes and env vars as described earlier in the doc

Follow existing doc conventions: use `---` separator before the new section, use h2 (`##`) for the main section, h3 (`###`) for subsections. All code blocks must have language identifiers (bash, yaml, json, text). Table separators use `| --- |` format with spaces.
