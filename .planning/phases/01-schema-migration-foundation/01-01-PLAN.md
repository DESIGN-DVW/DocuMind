<task type="auto">
  <name>Task 1: Create migration runner and bootstrap migration</name>
  <files>
    scripts/db/migrate.mjs
    scripts/db/migrations/001-schema-migrations-table.sql
  </files>
  <action>
Create `scripts/db/migrations/` directory.

Create `scripts/db/migrations/001-schema-migrations-table.sql`:
```sql
-- Bootstrap: create the migrations tracking table itself
CREATE TABLE IF NOT EXISTS schema_migrations (
  version TEXT PRIMARY KEY NOT NULL,
  applied_at TEXT NOT NULL,
  description TEXT
);
```

Create `scripts/db/migrate.mjs` as an ESM Node.js script with the following behavior:

1. **Backup** — Before opening the DB, copy `data/documind.db` to `data/documind.db.bak-{ISO-timestamp}` using `fs.copyFileSync`. Also copy `-wal` and `-shm` files if they exist (per Pitfall 2 in research). Use timestamp format with colons/dots replaced by dashes for filesystem safety.

2. **Open DB** — Use `better-sqlite3` with `pragma('foreign_keys = ON')` and `pragma('journal_mode = WAL')`. Use the same `__dirname` + `../../data/documind.db` path pattern as `init-database.mjs`.

3. **Bootstrap schema_migrations** — Run `CREATE TABLE IF NOT EXISTS schema_migrations (...)` directly in the runner (not just from the SQL file). This ensures the tracking table exists before attempting to read applied versions.

4. **Read applied versions** — `SELECT version FROM schema_migrations` into a Set.

5. **Read migration files** — Read all `.sql` files from `scripts/db/migrations/` directory, sorted lexicographically. The version is the filename without `.sql` extension (e.g., `001-schema-migrations-table`).

6. **Apply pending migrations** — For each unapplied migration:
   - Read the SQL file content
   - Wrap in `db.transaction()` — execute the SQL, then INSERT into schema_migrations with version, ISO timestamp, and filename as description
   - Log each applied migration with chalk.green
   - Log each skipped migration with chalk.gray

7. **Summary** — Print count of applied vs skipped migrations. Close DB.

8. **Error handling** — Wrap the entire runner in try/catch. On error, print chalk.red error message with stack trace and `process.exit(1)`. The transaction auto-rollback on throw handles per-migration failures.

9. **CLI output** — Print a header line with "DocuMind Database Migration" and the DB path. Print the backup path. Use the same chalk styling patterns as init-database.mjs (blue for info, green for success, gray for skipped, red for errors).

Do NOT call init-database.mjs from migrate.mjs. They are separate concerns: init creates a fresh DB from schema.sql; migrate evolves an existing DB.
  </action>
  <verify>
    <automated>node scripts/db/migrate.mjs 2>&1 | grep -q "schema_migrations" && echo "PASS" || echo "FAIL"</automated>
  </verify>
  <done>Running `node scripts/db/migrate.mjs` on the live DB creates a backup file, applies migration 001 (or skips it if already applied), and the schema_migrations table has a row for version "001-schema-migrations-table" with an applied_at timestamp.</done>

  <files>

  <action>
Create `scripts/db/reset-database.mjs` as an ESM Node.js script:

1. Check `process.argv` for `--force` flag.
2. If `--force` is NOT present:
   - Print a loud warning using chalk.red and chalk.yellow:
     ```text
     ╔══════════════════════════════════════════════════════════════╗
     ║  DESTRUCTIVE OPERATION: db:reset                           ║
     ║                                                             ║
     ║  This will DESTROY all indexed documents in documind.db     ║
     ║  Current corpus: 8,000+ documents                           ║
     ║                                                             ║
     ║  Run with --force to confirm:                               ║
     ║  npm run db:reset -- --force                                ║
     ╚══════════════════════════════════════════════════════════════╝
     ```
   - Exit with `process.exit(1)`.
3. If `--force` IS present:
   - Print warning that reset is proceeding.
   - Create a backup first (same pattern as migrate.mjs — copy .db, -wal, -shm files with timestamp).
   - Delete `data/documind.db`, `data/documind.db-wal`, `data/documind.db-shm` if they exist.
   - Run `init-database.mjs` logic: open new DB, read schema.sql, execute it, close. (Import the schema path, do not import init-database.mjs itself — just read and exec schema.sql directly to avoid the init script's console output mixing.)
   - Print confirmation that reset is complete.

Update `package.json` scripts — change these two entries:
- `"db:reset"`: from `"rm -f data/documind.db && npm run db:init"` to `"node scripts/db/reset-database.mjs"`
- `"db:migrate"`: from `"node scripts/db/init-database.mjs"` to `"node scripts/db/migrate.mjs"`

Leave `"db:init"` unchanged — it still creates a fresh DB from schema.sql for new installs.
  </action>

  </verify>
