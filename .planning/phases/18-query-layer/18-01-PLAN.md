### kuzuTraverseGraph(kuzuDb, docId, direction = 'forward', relType = null)

- Used by /graph REST endpoint for single-hop directional traversal

- direction='forward': `MATCH (src:Document {id: $id})-[r]->(tgt:Document)`

- direction='reverse': `MATCH (src:Document)-[r]->(tgt:Document {id: $id})`

- direction='both': Use UNION (not UNION ALL) of forward + reverse queries — avoid undirected `-` syntax since label(r) reliability on undirected is uncertain

- RETURN columns (must match exactly): `tgt.id AS doc_id, label(r) AS relationship_type, tgt.path AS path, tgt.repository AS repository, tgt.filename AS filename, tgt.category AS category, 1 AS depth`

- For reverse, swap src/tgt in RETURN: `src.id AS doc_id, label(r) AS relationship_type, src.path AS path, src.repository AS repository, src.filename AS filename, src.category AS category, 1 AS depth`

- Optional relType filter: when relType is provided, append `:${relType}` to `[r]` → `[r:imports]`

- Add `LIMIT 500` on each query branch

- Params: `{ id: docId }` (named params, not ? placeholders)

### kuzuFindRelated(kuzuDb, docId, maxDepth = 2, direction = 'forward')

- Used by get_related MCP tool for multi-hop traversal

- Start with `label(r[0])` approach on variable-length paths: `MATCH (src:Document {id: $id})-[r*1..$hops]->(tgt:Document) RETURN tgt.id AS doc_id, label(r[0]) AS relationship_type, 1.0 AS weight, length(r) AS depth, tgt.path AS path, tgt.repository AS repository, tgt.filename AS filename, tgt.category AS category ORDER BY depth, tgt.id LIMIT 200`

- direction='reverse': reverse the arrow direction in pattern

- direction='both': UNION forward + reverse variable-length queries

- Params: `{ id: docId, hops: maxDepth }`

- NOTE: label(r[0]) is LOW confidence — the smoke test in Task 2 will validate. If the smoke test fails, the executor must update this function to use the fallback: run a depth-1 single-hop query for relationship_type using `label(r)`, plus a separate reachability query, then merge results in JS. Document the chosen approach in a comment.

### Test sequence

1. Open `new kuzu.Database(KUZU_DIR)` — read-only queries only, no writes

2. Count Document nodes: `MATCH (d:Document) RETURN count(d) AS n` — if n=0, skip traversal tests and print "Graph empty — sync first with npm run graph:rebuild"

3. Find any document ID with at least one outgoing edge: `MATCH (src:Document)-[r]->(tgt:Document) RETURN src.id AS id LIMIT 1`

4. **Test A — single-hop forward:** `MATCH (src:Document {id: $id})-[r]->(tgt:Document) RETURN tgt.id AS doc_id, label(r) AS relationship_type, 1 AS depth LIMIT 5` — print first result row, confirm label(r) is a non-empty string

5. **Test B — label(r[0]) on variable-length path:** `MATCH (src:Document {id: $id})-[r*1..2]->(tgt:Document) RETURN tgt.id AS doc_id, label(r[0]) AS relationship_type, length(r) AS depth LIMIT 5` — this is the LOW confidence test

   - If Test B succeeds (no exception, relationship_type is a non-empty string): print "PASS: label(r[0]) works on variable-length paths — no fallback needed"

   - If Test B throws or returns empty relationship_type: print "FAIL: label(r[0]) does not work — FALLBACK REQUIRED: update kuzuFindRelated to use depth-1 union approach" and exit with process.exit(1)

6. **Test C — reverse direction:** `MATCH (src:Document)-[r]->(tgt:Document {id: $tgtId}) RETURN src.id AS doc_id, label(r) AS relationship_type LIMIT 3` where $tgtId is the target from Test A

7. Close result, conn, db. Print summary table.
