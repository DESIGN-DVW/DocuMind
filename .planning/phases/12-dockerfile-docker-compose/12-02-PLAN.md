<task type="auto">
  <name>Task 2: Create multi-stage Dockerfile</name>
  <files>Dockerfile</files>
  <action>
Create `Dockerfile` at the repository root with a two-stage build:

**Stage 1 — builder:**
- `FROM node:22-bookworm-slim AS builder`
- `WORKDIR /app`
- Install build tools: `apt-get update && apt-get install -y python3 make g++ && rm -rf /var/lib/apt/lists/*`
- `COPY package*.json ./`
- `RUN npm ci` (full install — native modules need all deps to compile)
- `RUN npm prune --omit=dev` (remove dev deps after native compilation is done — saves ~50-80MB)
- `COPY . .` (copies app code; .dockerignore prevents node_modules from host)

**Stage 2 — runtime:**
- `FROM node:22-bookworm-slim AS runtime`
- `WORKDIR /app`
- Install runtime tools only: `apt-get update && apt-get install -y dumb-init curl && rm -rf /var/lib/apt/lists/*`
- Create non-root user: `RUN groupadd -r documind && useradd -r -g documind -d /app documind`
- Create data directory owned by documind: `RUN mkdir -p /app/data && chown documind:documind /app/data`
- `COPY --from=builder /app/node_modules ./node_modules`
- `COPY --chown=documind:documind . .`
- Safety: `RUN rm -f data/documind.db data/*.db-wal data/*.db-shm .env`
- `USER documind`
- `EXPOSE 9000`
- HEALTHCHECK: `HEALTHCHECK --interval=15s --timeout=5s --start-period=30s --retries=3 CMD curl -f http://localhost:9000/health || exit 1`
- `ENTRYPOINT ["dumb-init", "--"]`
- `CMD ["node", "daemon/server.mjs"]`

**Anti-patterns to avoid (do NOT do these):**
- Do NOT use `node:22-alpine` — musl libc breaks better-sqlite3
- Do NOT use `CMD ["npm", "start"]` — npm swallows SIGTERM
- Do NOT use PM2 in the container — exits immediately
- Do NOT add `ENV GITHUB_TOKEN` or any secrets — visible in docker history
- Do NOT use `node:24` — N-API incompatibility with better-sqlite3
  </action>

<task type="auto">
  <name>Task 3: Create docker-compose.yml and build + verify the image</name>

  <action>
**Part A — Create docker-compose.yml:**

Create `docker-compose.yml` at the repository root:

```yaml
services:
  documind:
    build: .
    ports:
      - "${DOCUMIND_PORT:-9000}:9000"
    volumes:
      - documind_data:/app/data
      # Mount host repos read-only. Adjust the left side to your repos path.
      # Example: /Users/Shared/htdocs/github/DVWDesign:/repos:ro
    environment:
      DOCUMIND_REPOS_DIR: /repos
      CHOKIDAR_USEPOLLING: "true"
      CHOKIDAR_INTERVAL: "2000"
    restart: unless-stopped

volumes:
  documind_data:
```

**Key decisions:**
- Named volume `documind_data:/app/data` — NOT a bind mount (WAL mode safety on Docker Desktop for Mac)
- Repo bind mount is COMMENTED OUT — user must set the left side to their actual repos path. This avoids hardcoding `/Users/Shared/htdocs/github/DVWDesign` which only works on the developer's machine
- `CHOKIDAR_USEPOLLING=true` with 2-second interval for Docker Desktop VirtioFS compatibility
- `DOCUMIND_REPOS_DIR=/repos` tells the daemon to look for repos at the mount point
- Port is configurable via DOCUMIND_PORT env var (defaults to 9000)
- No healthcheck in compose — it is already in the Dockerfile HEALTHCHECK instruction (no duplication)
- `restart: unless-stopped` so daemon survives host reboots

**Part B — Build the image and verify:**

1. Run `docker build -t documind:dev .` and confirm:
   - Build context is under 10MB (check "transferring context" line)
   - Build completes without errors
   - `docker images documind:dev --format "{{.Size}}"` shows under 400MB

2. Run `docker run --rm documind:dev whoami` and confirm output is `documind` (non-root)

3. Start the container to test health + shutdown:
   - `docker compose up -d`
   - Wait up to 30 seconds, then `curl -f http://localhost:9000/health` should return 200
   - `time docker stop $(docker compose ps -q)` should complete in under 5 seconds
   - Check exit code: `docker inspect $(docker compose ps -q) --format='{{.State.ExitCode}}'` should be 0

4. Test volume persistence:
   - `docker compose up -d` again
   - Confirm /health returns 200 (data volume persisted)
   - `docker compose down` (stops containers but preserves volumes)

If the build context is over 10MB, revisit .dockerignore. If the image is over 400MB, check whether npm prune --omit=dev ran correctly in the Dockerfile.
